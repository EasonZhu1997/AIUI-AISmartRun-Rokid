# AISmartRun Backend and EverMind Contract

Date: 2026-07-08

## Purpose

AISmartRun keeps the AI answer path on Rokid AIUI. The glasses app uses the official AIUI `LanguageModel` capability, while the backend owns EverMind configuration, memory retrieval, anonymous device auth, and answer record storage.

No EverMind secret, provider key, or workspace identifier should be shipped inside the AIUI package.

## Default Client

- `app_id`: `AISmartRun` (client identifier)
- `app_key`: shared anonymous-login key, **required by the deployed backend** (`422` when missing, `401` when mismatched)
- `device_id`: stable anonymous device id generated by the glasses app
- Default backend: `https://119.28.104.126.nip.io`

The backend maps the authenticated anonymous user to its server-side EverMind configuration.

`app_key` provisioning on glasses: the public repo does not hard-code the key. It is injected into
`wx storage` under `coach_app_key` during debug/provisioning (or baked in once the repo is private).
When no `app_key` is present, the glasses client skips anon-login entirely (no doomed request, no
timeout cost) and the memory chain is explicitly off — the coach still answers via the official
AIUI `LanguageModel` plus the rule fallback.

## Endpoints

| Endpoint | Called by | Purpose |
|---|---|---|
| `POST /api/coach-svc/coach/anon-login` | AIUI, APK | Exchange `app_key` + `device_id` for a short-lived JWT (`app_id` accompanies as client identifier). The backend decides EverMind tenant/workspace routing. |
| `POST /api/coach-svc/coach/memory-context` | AIUI | Retrieve compact EverMind/local memory context. This endpoint must not run the main LLM for AIUI. |
| `POST /api/coach-svc/coach/aiui-record` | AIUI | Store the AIUI-generated question/answer turn and double-write to EverMind when enabled. |
| `POST /api/coach-svc/coach/chat` | APK | APK-compatible path where backend DeepSeek can generate and store the answer. AIUI does not use this as its primary answer path. |
| `POST /api/coach-svc/runs` | AIUI, APK | Store a run summary. AIUI sends aggregate metrics only (`source="aiui"`, no track/GPS) after a run ends; the backend's existing post-run insight/weekly-report pipeline picks it up with zero backend changes. Same anon JWT as the coach chain; best-effort with a local pending queue (cap 5) flushed silently on the home page. |

## Request Shape

Anonymous login:

```json
{
  "app_id": "AISmartRun",
  "app_key": "<provisioned shared key>",
  "device_id": "aiui-k2p4..."
}
```

The deployed backend (`server/coach/app/api/coach.py`) requires `app_key` as a mandatory field —
requests without it fail validation (`422`) and mismatches return `401`. The `app_key` is a shared
anonymous-login gate (rate-limited per device), not an EverMind secret; EverMind credentials never
leave the backend. History note: an earlier revision of this contract claimed "route by `app_id`
only" — that behavior was never implemented server-side and the claim caused a silent integration
break on the glasses; this document now reflects the deployed reality.

Memory retrieval:

```json
{
  "query": "心率高吗"
}
```

Record write:

```json
{
  "message": "[实时 心率 156(Z4) 配速 5:42/km] 心率高吗",
  "reply": "放慢一点，稳住呼吸。",
  "source": "aiui+evermind"
}
```

## Failure Rules

- Backend timeout or auth failure must not block AIUI `LanguageModel` answers.
- Memory retrieval is best-effort; empty memory means the official model still answers from live run data.
- Record write is best-effort; failure should be silent on glasses.
- EverMind provider errors should be handled and logged server-side.

## Environment Ownership

Backend should own:

- EverMind API key and workspace routing
- DeepSeek/API keys for the APK-compatible `/coach/chat` endpoint
- token TTL and refresh behavior
- double-write policy for AIUI records
- staging/production endpoint selection

AIUI package should own:

- stable anonymous `device_id`
- official AIUI `LanguageModel` prompt and short answer behavior
- best-effort memory context request
- best-effort record write

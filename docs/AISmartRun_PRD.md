## 1. 产品概述

AISmartRun 是运行在 Rokid 眼镜上的轻量跑步助手。用户打开后先看到正式首页：单眼镜模式已就绪，首页只读取设置与已记住设备并显示心率就绪状态，不建立蓝牙连接；点“开跑”后进入跑步 HUD，由 HUD 完成心率设备的扫描与连接（已记住设备优先），并用大号绿色数据展示时长、步频、估算距离和配速。当标准蓝牙心率设备接入并收到真实心率后，同一面板补充心率和心率区间。用户也可以进入蓝牙设备页搜索、记住或清除首选心率设备。跑步过程中，用户可以向 AI 教练询问配速、心率和节奏，教练会优先使用 Rokid 官方 AIUI `LanguageModel` 能力调用 DeepSeek，并结合当前实时跑步数据给出一句简短语音建议；EverMind 用于后台记忆检索和问答记录存储。

产品定位不是完整运动手表，也不是复杂训练 App，而是“戴着眼镜跑步时，最少操作、最少干扰、最快看到关键数据”的跑步 HUD 与语音教练。

## 2. 背景与问题

跑步时用户不适合频繁低头看手机或手表，也不适合阅读长文本。眼镜端的价值在于把少量关键数据固定在视野中，并用语音处理需要解释或判断的问题。

当前项目经历过模式选择、蓝牙/无蓝牙入口等方案后，已收敛为“就绪状态首页 + 一键开跑”的极简形态：不要求用户先选择室内/户外、蓝牙/无蓝牙；首页显示单眼镜已就绪和心率就绪状态，但不建立蓝牙连接；心率作为可选增强数据源由跑步 HUD 在开跑后自动接入（已记住设备优先），失败不阻断跑步。

## 3. 目标与非目标

### 3.1 产品目标

1. 打开 AISmartRun 后进入正式首页，明确显示单眼镜已就绪和心率就绪状态；首页不建立蓝牙连接，连接由跑步 HUD 在开跑后完成。
2. 单眼镜模式下也能显示可用数据：时间、步频、估算距离、配速。
3. 心率设备接入后在同一 HUD 补充心率，不切换页面。
4. 跑步 HUD 只做展示，不放点击按钮，降低跑步中误触。
5. AI 教练读取当前跑步快照，用短语音回答节奏、安全和配速问题。
6. 用户点“开跑”后进入跑步 HUD，并自动开始记录。
7. 提供蓝牙设备页，让用户搜索、记住、清除首选心率设备，并控制自动心率开关；首选设备只在设备页显式配对时写入。
8. 提供独立跑前配置页，让用户调整步长、自动心率、语音提示和记忆增强。
9. BLE、IMU、AI、网络任一能力不可用时，产品仍能降级为可用状态。

### 3.2 非目标

1. 不做完整训练计划、社交排行、地图轨迹和赛后分析。
2. 不承诺 GPS 级距离精度；无 GPS API 时距离由步频和估算步长计算。
3. 不提供医疗诊断或疾病建议；心率异常只做运动安全提醒。
4. 不接入私有手表协议；MVP 只承诺标准 Bluetooth Heart Rate Service。
5. 不在跑步 HUD 中提供复杂设置、设备列表或历史记录入口。

## 4. 目标用户

核心用户是使用 Rokid Glasses 跑步、慢跑或轻运动的人：

- 希望快速开跑，不想在运动前配置模式。
- 跑步时只需要看清几个关键指标。
- 可能没有心率带或运动手表，但希望眼镜能独立记录基础数据。
- 有标准蓝牙心率设备时，希望心率自然出现在同一面板。
- 希望在跑步中用语音问“现在快不快”“心率高不高”“要不要降速”。

## 5. MVP 范围

### 5.1 页面范围

| 页面 | 路径 | 定位 |
|---|---|---|
| 首页 | `pages/index/index` | 默认入口，显示单眼镜已就绪和心率就绪状态（不建立连接），并提供开跑/设备 |
| 跑步 HUD | `pages/run_hud/index` | 开跑后进入，自动记录并展示实时跑步数据 |
| 蓝牙设备页 | `pages/bluetooth/index` | 搜索、记住和清除首选心率设备，控制自动连接 |
| 设置页 | `pages/settings/index` | 跑前调整基础偏好，不进入跑步 HUD 主流程 |
| AI 教练 | `pages/coach/index` | 语音问答，回答配速、心率和节奏问题 |

### 5.2 数据范围

| 数据 | 来源 | MVP 规则 |
|---|---|---|
| 时长 | `RunSession` 本地计时 | 进入 HUD 自动开始；息屏/切页自动暂停记录，回来自动继续，时长与距离口径一致 |
| 步频 | 眼镜 Accelerometer + `StepDetector` | IMU 可用时估算，无法使用时显示占位 |
| 距离 | 步频 x 估算步长积分 | 粗估，仅用于跑步中参考；暂停期间不累计 |
| 配速 | 估算速度 | 静止（步频为 0）或速度无效时显示 `--:--`，不回退为全程平均 |
| 心率 | 标准 BLE HRS notify | 只有收到有效 bpm 后才显示心率列 |
| 教练上下文 | `lib/live.js` 实时快照 | 无实时数据时不编造数字 |

## 6. 用户流程

### 6.1 打开首页

1. 用户打开 AISmartRun。
2. `app.json` 默认进入 `pages/index/index`。
3. 首页只读取设置与已记住设备，显示单眼镜已就绪和心率就绪状态，不建立蓝牙连接：
   - 自动心率开关关闭 → 显示“已关闭”。
   - 宿主无蓝牙能力 → 显示“不可用”。
   - 有已记住设备 → 显示“已记住，开跑自动连”。
   - 无已记住设备 → 显示“待连接，设备页可配对”。
4. 任一状态下都可开跑；真正的扫描与连接由跑步 HUD 在开跑后完成。
5. 用户可选择“开跑”进入跑步 HUD，或选择“设备”进入蓝牙设备页。
6. 首页语音唤醒等同于“开跑”，与一键开跑主线一致。

### 6.2 蓝牙设备配置

1. 用户在首页点击“设备”进入 `pages/bluetooth/index`。
2. 设备页可搜索标准心率设备。
3. 搜到设备后验证其标准心率服务，并保存为首选设备；首选设备只在设备页显式配对时写入。
4. 用户可关闭自动心率开关或清除首选设备。
5. 首选设备用于跑步 HUD 开跑后的自动连接优先级。

### 6.3 一键开跑

1. 用户在首页点击“开跑”，或用 Enter / Space / GlobalHook / 语音唤醒触发默认动作。
2. 进入 `pages/run_hud/index`。
3. HUD `onLoad` 创建跑步会话并开始计时。
4. HUD 自动扫描标准心率设备，优先连接已记住的首选设备；找不到首选时兜底连接第一个可用设备，但兜底设备不写入首选。
5. 未接入心率时保持单眼镜模式；接入并收到 bpm 后切换为心率同屏布局。

### 6.4 单眼镜跑步

1. 用户没有心率设备，或自动扫描失败。
2. HUD 显示配速、步频、距离、时长四项。
3. 页脚数据源角标显示短标签“眼镜估算”；传感器不可用时显示“仅计时”。
4. 不显示心率列，也不显示 `--` 心率占位。

### 6.5 心率接入

1. 跑步 HUD 在开跑后自动扫描支持 Heart Rate Service 的蓝牙设备（首页不扫描）。
2. 若有首选设备，优先等待并连接首选设备；否则兜底连接第一个可用心率设备，兜底设备不写入首选。
3. 收到有效心率 notify 后，HUD 同一卡片扩展为心率、配速、步频、距离、时长，数据源角标显示“心率+眼镜”。
4. 左侧点阵用单绿色透明度表达心率区间，不使用红、橙等第二色；心率低于 50% maxHr 时不点亮任何区间。
5. GATT 断连或连续 8 秒无新心率数据时，HUD 静默回退单眼镜模式，跑步不中断。

### 6.6 AI 教练问答

1. 用户进入教练页后，语音唤醒等同于“开始提问”（与首页语音唤醒=开跑区分）。
2. 教练读取 `run_hud` 写入的实时快照；快照超过 10 秒视为过期，按“暂无运动数据”处理。
3. ASR 识别用户问题。
4. 心率处于 Z5 时直接使用确定性安全规则回答（不经过 LLM），保证安全提醒即时且稳定。
5. 其余问题优先使用 Rokid 官方 AIUI `LanguageModel` 能力生成短答；LLM 会话 system prompt 只含教练人设，实时快照在每轮问题时注入。
6. LLM 流式回答设 10 秒总超时，超时则切本地规则兜底；LLM 输出做消毒（去 markdown 和换行、截断到一句话）后再展示和播报。
7. 开启记忆增强时，先从后端取 EverMind/本地记忆上下文并注入本轮问题；记忆/登录请求超时 2.5 秒，登录失败后 60 秒内不重试，失败不阻断回答。
8. 官方模型成功回答后，AIUI 端调用后端记录接口写入聊天历史，并由后端双写 EverMind。
9. TTS 可用时播放一句短建议。

### 6.7 跑前配置

1. 用户打开设置页。
2. 可调整估算步长、自动心率、语音提示和记忆增强。
3. 设置写入本地 `wx storage`。
4. 下次进入跑步 HUD 或教练页时读取最新设置。
5. 设置页可直接进入跑步 HUD；应用默认入口仍是正式首页。

## 7. 功能需求

### FR-1 默认进入正式首页

优先级：P0

需求：
- `app.json` 第一个页面必须是 `pages/index/index`。
- 首页显示单眼镜已就绪和心率就绪状态；就绪状态只由设置与已记住设备推导（已关闭 / 不可用 / 已记住 / 待连接），不建立蓝牙连接。
- 首页提供“开跑”和“设备”两个动作。
- Enter / Space / GlobalHook 默认触发开跑；首页语音唤醒等同于开跑。
- Backspace 在首页触发退出应用，由页面显式处理，不依赖宿主默认行为。

验收：
- 打开应用后首屏是正式首页。
- `pages/index/index.ink` 使用 `home-card`。
- `pages/index/index.ink` 包含 `openRun()`、`openBluetooth()` 和 `refreshHeartReadiness()`；不再包含 `autoConnectBle()`。
- 首页显示上述四种心率就绪状态之一。
- 首页不使用 `<button>`。

### FR-2 跑步 HUD 自动记录

优先级：P0

需求：
- 从首页开跑后进入 HUD。
- HUD 进入后自动开始自由跑。
- 用户不需要再点击开始、选择模式或选择设备。

验收：
- `pages/run_hud/index.ink` 的 `onLoad()` 会启动跑步会话。
- 跑步 HUD 内不包含按钮。

### FR-3 单眼镜模式

优先级：P0

需求：
- 无外设时展示四个核心数据：配速、步频、距离、时长。
- 使用眼镜加速度计估算步频和距离。
- 心率缺失时不渲染心率 UI。
- 加速度计不可用时降级为计时可用状态。

验收：
- `showHeartRate=false` 时渲染 `glasses-grid`。
- 四列布局没有心率占位。
- IMU 不可用不导致页面崩溃。

### FR-4 标准蓝牙心率接入

优先级：P0

需求：
- 支持 Bluetooth Heart Rate Service：`0x180D / 0x2A37`。
- 只有跑步 HUD 在开跑后自动扫描并尝试连接心率设备；首页不发起扫描或连接。
- 若用户保存了首选心率设备，自动连接优先匹配首选设备；找不到首选时兜底连接第一个可用设备，但兜底设备不写入首选（首选只在设备页显式配对时写入，防止邻近跑者的心率带被永久记住）。
- 蓝牙扫描、连接或 notify 失败时静默留在单眼镜模式。
- GATT 断连或连续 8 秒无新心率数据时，静默回退单眼镜模式。
- 只有收到有效 bpm 后才显示心率同屏布局。

验收：
- HUD 的自动连接使用 heart_rate service filter。
- 自动连接逻辑使用首选设备匹配，找不到首选时回退到第一个可用设备，且不把兜底设备写为首选。
- 心率数据由 `parseHeartRateMeasurement()` 解析。
- `hasHeartRate = connected && bpm 有效` 时才设置 `showHeartRate=true`。
- 断连或 8 秒无新数据后 `showHeartRate` 回到 `false`，跑步会话不中断。

### FR-5 跑步数据刷新与清理

优先级：P0

需求：
- HUD 每 1 秒聚合刷新一次。
- 数据刷新统一使用 `setData`。
- 息屏/切页（`onHide`）时自动暂停记录并停止 ticker 和 accelerometer；暂停期间时长与距离都不累计，两者口径一致。
- 页面重新显示（`onShow`）且仍在跑步时自动继续记录并恢复 ticker 和 accelerometer，不出现双倍跳秒。
- 页面卸载时清理 BLE、ticker、accelerometer 和实时快照。

验收：
- `TICK_MS = 1000`。
- `onHide()` 自动暂停记录并停止资源。
- `onShow()` 在 running 状态下自动继续记录并恢复资源。
- 暂停期间时长不增加；恢复后不追补暂停时间。
- `onUnload()` 清理所有运行时资源。

### FR-6 AI 教练

优先级：P1

需求：
- 支持语音唤醒或手动触发 ASR；教练页语音唤醒等同于开始提问。
- 教练回答必须参考实时跑步快照；快照超过 10 秒视为过期，按无数据处理。
- 心率进入 Z5 时直接使用确定性安全规则回答，不经过 LLM。
- 其余问题优先使用 Rokid 官方 AIUI `LanguageModel` 能力生成回答，EverMind 仅承担记忆检索和记录存储。
- LLM 会话 system prompt 只含教练人设；实时快照在每轮问题时注入，不固化在会话 prompt 中。
- LLM 流式回答设 10 秒总超时，超时切规则兜底。
- LLM 输出必须消毒：去 markdown 和换行、截断到一句话。
- 后端记忆/登录请求超时 2.5 秒；登录失败后 60 秒内不重试。
- 没有实时数据时明确按“暂无运动数据”处理，不编造心率、距离或配速。
- 回答必须短、口语化、适合 TTS。
- 模型不可用时必须有规则兜底。

验收：
- 教练页读取 `readLiveSnapshot(wx)`；过期快照返回空。
- Z5 快照下的回答来自确定性安全规则，不调用 LLM。
- LLM 超时或失败时 `fallbackCoachReply()` 覆盖配速、心率、距离、时间和通用问题。
- LLM 输出经消毒后为单句纯文本。
- ASR 不可用时页面给出可理解状态。
- TTS 不可用不阻断文本回答。
- 后端不可达时回答延迟不超过“2.5 秒记忆超时 + LLM 回答”的量级，不出现长时间卡“思考”。

### FR-7 跑前配置页

优先级：P1

需求：
- 提供独立设置页，不在跑步 HUD 内放配置控件。
- 支持调整估算步长。
- 支持关闭自动心率扫描。
- 支持关闭语音播报，仅保留文本提示。
- 支持关闭记忆增强请求。
- 设置持久化到本地 storage。

验收：
- `app.json` 包含 `pages/settings/index`。
- 设置页使用 `settings-card` 卡片式布局。
- 跑步页读取 `readRunSettings(wx)`。
- 教练页读取 `readRunSettings(wx)`。

### FR-8 蓝牙设备页

优先级：P1

需求：
- 提供独立设备页，不在跑步 HUD 内放设备列表。
- 支持搜索标准蓝牙心率设备。
- 搜到设备后保存为首选心率设备；这是首选设备的唯一写入入口。
- 支持清除首选设备。
- 支持控制“自动心率”开关（行名统一为“自动心率”）。

验收：
- `app.json` 包含 `pages/bluetooth/index`。
- 设备页使用 `device-card` 卡片式布局。
- 首选设备写入本地 storage，且只有设备页显式配对时写入。
- 跑步 HUD 自动连接时读取首选设备；首页只读取用于显示就绪状态。

### FR-9 主动语音提示

优先级：P1

需求：
- 跑步中可根据状态主动播报安全或节奏提示。
- 优先级为：Z5 安全降速、整公里、每 5 分钟、进入 Z4。
- 停留在 Z5 期间每满 1 分钟重复播报一次降速提醒，直到离开 Z5（不是只在进入瞬间提醒一次）。
- 主动提示必须短，不打断数据展示。

验收：
- `nextProactiveCue()` 输出一句短提示或 `null`。
- Z5 提示优先于里程碑提示。
- 持续处于 Z5 时每满 1 分钟再次输出降速提醒。
- 没有触发条件时不播报。

### FR-10 防误触退出

优先级：P0

需求：
- 跑步中按一次 Backspace 不直接结束跑步。
- 第一次按下后 HUD 显示“再按一次结束”提示。
- 3 秒内再按一次 Backspace 才结束跑步、释放资源并返回。
- 3 秒内未按第二次则提示消失，跑步继续，数据不丢失。

验收：
- HUD 单击 Backspace 后跑步会话仍在记录。
- 3 秒窗口内双击 Backspace 才触发资源清理和返回。
- 超时后再次单击重新进入确认流程。

### FR-11 跑步记录上传（best-effort）

优先级：P1

需求：
- 跑步结束（确认退出或页面卸载）时，把本次跑步的汇总指标写入后端 `runs` 表（`source="aiui"`），让眼镜用户复用姊妹 APK 生态既有的跑后 AI 点评、洞察和周报管线。
- 只上传汇总指标：开始/结束时间、净时长、估算距离、平均配速、平均/最高心率、平均步频。不上传轨迹、不上传 GPS。
- 上传门槛：时长不足 60 秒且距离不足 100 米的会话不上传，不制造垃圾记录。
- 退出时只写入本地待传队列（上限 5 条，FIFO 丢最老）；真正的网络发送在首页 `onLoad` 静默执行，不阻塞任何交互。
- 鉴权复用教练页同一枚匿名 JWT（`coach_app_key` 注入口径）；无 key 时不发请求，队列保留，等 key 注入后自然补传。
- 上传失败（断网/超时/后端异常）静默保留队列；token 过期（401）清 token 后保留，下次重新直登再传。
- 一次跑步会话只入队一次（确认退出与页面卸载不得重复入队）。

验收：
- `run_hud` 确认退出后 `pending_run_uploads` 有且仅有一条 `source="aiui"` 记录。
- 门槛以下的会话不入队。
- 首页无 `coach_app_key` 时零网络请求且队列保留。
- 首页有 key 时完成匿名直登与上传，成功后清队；401 清 token 保留队列。

## 8. UI 与交互规范

### 8.1 全局视觉

- 目标画布：480px 宽，当前卡片高度 150px。
- 背景：黑色。
- 主题：单绿色 token。
- 卡片：2px 绿色边框，12px 圆角。
- 禁止 emoji。
- 禁止使用蓝、红、橙等第二色；警告和异常通过绿色透明度、边框粗细、点阵数量表达。
- 跑步页不放按钮，只展示数据和必要状态。

### 8.2 正式首页

- 顶部：跑步图标、`AISmartRun`、就绪状态。
- 主体：自由跑主块、单眼镜已就绪、心率就绪状态（已关闭 / 不可用 / 已记住 / 待连接）。
- 底部：无心率也可开跑、设备、开跑。
- 默认焦点为“开跑”。

### 8.3 蓝牙设备页

- 顶部：图标、`设备` 标题、设备状态。
- 主体：首选心率、搜索设备、自动心率、忘记设备四行。
- 当前焦点通过更粗绿色边框表达。
- 支持方向键移动，Enter / Space / GlobalHook 修改当前项。
- Backspace 停止扫描/连接清理后返回上一页。
- 页面底部提供 `设置` 和 `开跑` 动作。

### 8.4 跑步 HUD

单眼镜模式：
- 顶部：跑步图标 + `单眼镜模式`。
- 主体：配速、步频、距离、时长四列。
- 页脚：教练短句 + 数据源角标。

心率接入模式：
- 顶部：跑步图标 + `心率接入`。
- 主体：心率、配速、步频、距离、时长。
- 左侧：心率区间点阵。
- 页脚：教练短句 + 数据源角标。

数据源角标使用短标签：`心率+眼镜`（心率已接入）、`眼镜估算`（仅 IMU）、`仅计时`（传感器不可用）。

Backspace 采用双击确认：第一次按下显示“再按一次结束”，3 秒内再按一次才释放 ticker、加速度计、蓝牙和实时快照并返回上一页；超时则继续跑步。

### 8.5 AI 教练页

- 顶部：图标、`教练` 标题、状态 chip。
- 中部：单气泡逐句显示用户提问或教练回复。
- 底部：可询问主题 + 手动触发控件。
- 回复文案保持一到两行，避免长文本阅读负担。
- Backspace 在语音/思考/播放中取消当前轮次，否则返回上一页。

### 8.6 设置页

- 顶部：图标、`设置` 标题、保存状态。
- 主体：步长、自动心率、语音提示、记忆增强四行。
- 当前焦点通过更粗绿色边框表达。
- 支持方向键移动，Enter / Space / GlobalHook 修改当前项。
- Backspace 返回上一页。
- 页面底部提供 `开跑` 动作，进入跑步 HUD。

## 9. 权限与依赖

| 权限 | 用途 |
|---|---|
| `bluetooth` | 连接标准心率手表、心率带和跑步设备 |
| `accelerometer` | 无外设时估算步数、步频、距离和配速 |
| `microphone` | 接收跑步中的语音提问 |
| `audio` | 播放 AI 教练语音建议 |
| `network` | 获取历史跑步记忆，并把 AIUI 官方模型生成的问答写入后端和 EverMind |

运行时能力：
- `navigator.bluetooth.scanDevices`
- `Accelerometer`
- `SpeechRecognition`
- `LanguageModel`
- `wx.speech.playTTS` 或 Web Speech TTS
- `wx storage`

## 10. 数据与隐私

- MVP 不采集 GPS 轨迹。
- 心率数据只作为实时运动上下文使用。
- 跑步快照通过本地 `wx storage` 在 HUD 和教练页之间传递；快照带写入时间戳，读取时超过 10 秒视为过期并返回空，教练不会把旧数据当作“此刻”。
- 结束跑步时清理实时快照，避免教练读取上一段运动数据。
- EverMind 用于记忆上下文和问答记录存储；AIUI 端默认传 `app_id=AISmartRun` 和匿名设备 ID，密钥、空间和双写策略由后端配置；网络或鉴权失败时不影响设备端/本地兜底建议。
- 跑步记录上传只含汇总指标（时长/距离/配速/心率/步频），不含轨迹与 GPS；归属后端匿名用户，删除跟随后端 PIPL 注销链路。
- AI 教练不得输出医疗诊断。

## 11. 非功能需求

### 11.1 可用性

- 从打开应用到看到 HUD 应尽量小于 1 秒。
- 无 BLE、无 IMU、无 ASR、无 LLM 时都要可降级。
- 跑步 HUD 内不出现需要精细点击的控件。

### 11.2 稳定性

- BLE 失败不阻断跑步会话。
- 所有协议解析函数对非法包返回空结果或占位，不抛出未捕获异常。
- 页面隐藏、卸载、恢复时不得产生重复 ticker 或重复传感器监听。

### 11.3 性能

- HUD 刷新频率为 1Hz。
- 每次 tick 合并数据后一次 `setData`。
- EverMind 记忆/登录请求超时为 2.5 秒，登录失败后 60 秒内不重试，不阻断官方 AIUI 模型回答。
- LLM 流式回答总超时 10 秒；官方模型不可用或超时进入本地规则兜底链路。

### 11.4 可读性

- 主数据使用大字号。
- 跑步页只保留必要状态，不放小号说明字。
- 文案必须短，语音提示必须能在跑步中快速理解。

## 12. 验收标准

### 12.1 自动化验收

执行：

```bash
npm test
```

必须满足：
- 跑步会话、格式化、心率解析、IMU、实时快照、教练兜底等核心纯逻辑测试通过。
- 失败用例必须在提测前修复。

### 12.2 页面验收

- `app.json` 首个页面为 `pages/index/index`。
- 首页显示心率就绪状态（不建立连接），并可进入设备页或跑步 HUD。
- 设备页可搜索、记住、清除首选心率设备。
- `run_hud` 初始展示单眼镜四列布局。
- 设置页可调整并保存跑前偏好。
- 无心率时不出现心率列。
- 有有效心率时同一卡片显示心率列和区间点阵。
- HUD 内不包含按钮。
- 教练页能在无实时数据时给出不编造数字的回答。

### 12.3 真机验收

必须在 Rokid 眼镜上验证：
- 眼镜端进入 HUD 后自动开始计时。
- 加速度计步频估算能随跑步动作变化。
- Fenix 8 或 ESP32 HRS 模拟器可广播并接入心率。
- 息屏、浮层或页面切换时自动暂停记录，回来后自动继续，无双倍跳秒。
- 跑步中 Backspace 双击确认生效，单击不丢跑步数据。
- ASR、LLM、TTS 可用性与失败兜底符合预期。
- 1Hz 刷新下功耗和流畅度可接受。

## 13. 风险与待确认

| 风险 | 影响 | 当前策略 |
|---|---|---|
| BLE 扫描可能需要用户手势 | 自动接入失败 | 静默留在单眼镜模式，真机验证后决定是否增加入口外触发 |
| IMU 估算距离误差 | 配速和距离不够精确 | 文案和定位明确为估算，后续可接 RSC/FTMS |
| 眼镜后台存活不确定 | 息屏后数据冻结或 BLE 断开 | onHide 自动暂停记录、onShow 自动继续；BLE notify 行为仍需真机验证 |
| maxHr 固定 190 | 心率区间对年长/年轻用户不准 | 已知限制；低于 50% maxHr 不点亮区间；“按年龄/自定义最大心率”列入 P1 路线图 |
| ASR / LLM / TTS 宿主不可用 | 教练体验下降 | 文本状态 + 本地规则兜底 |
| 英文文案过长 | HUD 挤压 | 英文版使用短标签：HR、Cad、Dist |
| 设置入口过深 | 用户不易发现配置 | 设置经设备页进入，Alpha 观察可发现性 |

## 14. 后续路线
